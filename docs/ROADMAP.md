# Mova 功能现状与开发路线

这份文档只描述当前代码里已经落地的能力，以及接下来继续实现时最值得优先投入的部分。

如果你关心当前工程结构是否合理、哪些模块应该拆、目录后续该如何收敛，见 [`docs/ARCHITECTURE.md`](/Users/riches/Desktop/mova/docs/ARCHITECTURE.md)。

## 1. 当前已实现的能力

### 基础服务

- `mova-server` 可启动 Axum HTTP 服务。
- 启动时会自动连接 PostgreSQL、执行 migration、再做一次 `ping`。
- `GET /api/health` 会同时检查服务进程和数据库可用性。
- `mova-web` 已提供基于 React + Vite 的前端原型，可直接联调现有 API。

### 用户与认证

- 支持查询 bootstrap 状态 `GET /api/auth/bootstrap-status`。
- 当系统里还没有管理员时，支持初始化首个管理员 `POST /api/auth/bootstrap-admin`。
- 支持登录、登出和当前用户查询 `POST /api/auth/login` / `POST /api/auth/logout` / `GET /api/auth/me`。
- 当前用户角色区分为 `admin` / `viewer`。
- `admin` 默认拥有全部媒体库访问权；`viewer` 只可见被分配的媒体库。
- 已支持管理员查询和创建用户 `GET /api/users` / `POST /api/users`。
- 已支持管理员更新用户基础信息、删除用户和重置密码 `PATCH /api/users/{id}` / `DELETE /api/users/{id}` / `PUT /api/users/{id}/password`。
- 已支持为普通用户更新媒体库访问范围 `PUT /api/users/{id}/library-access`。
- 已支持当前登录用户修改自己的密码 `PUT /api/auth/password`，修改后会轮换 session。
- 除健康检查和 bootstrap / 登录相关接口外，现有浏览、播放和管理接口都已经挂到 session 鉴权下。

### 媒体库管理

- 支持创建媒体库 `POST /api/libraries`。
- 支持查询媒体库列表 `GET /api/libraries`。
- 支持查询媒体库详情 `GET /api/libraries/{id}`。
- 支持更新媒体库基础配置 `PATCH /api/libraries/{id}`。
- 支持删除媒体库 `DELETE /api/libraries/{id}`。
- 媒体库类型当前支持 `mixed` / `movie` / `series`，其中 `mixed` 会按单文件自动判断电影或剧集。
- 创建媒体库时会校验 `root_path` 是否存在且为目录。
- 新建且启用的媒体库会自动触发第一次后台扫描。
- 已支持在更新媒体库时直接启停 watcher 和自动同步；重新启用时会补一轮后台扫描。
- 删除媒体库时会先阻止新的扫描进入，并等待当前扫描安全停止后再执行删除。

### 扫描任务

- 支持为媒体库创建异步扫描任务 `POST /api/libraries/{id}/scan`。
- 支持查询扫描历史 `GET /api/libraries/{id}/scan-jobs`。
- 支持查询单个扫描任务状态 `GET /api/libraries/{id}/scan-jobs/{scan_job_id}`。
- 扫描任务状态目前包含 `pending / running / success / failed`。
- 扫描发现文件时会实时更新 `scan_jobs.scanned_files`。
- 支持 `GET /api/events` 的 SSE 实时事件流，当前会主动广播 `scan.job.updated`、`scan.job.finished`、`scan.item.updated`、`library.updated`、`library.deleted` 和 `media_item.metadata.updated`。
- 扫描实时事件已补上 `discovering / enriching / syncing / finished` 阶段，并会在发现新文件时立刻推送条目级占位信息；同一张卡会继续经历元数据、图片和等待写库阶段。
- 扫描失败时，`scan_job.error_message` 现在会直接带阶段上下文，例如扫描目录失败、元数据补全失败、写库失败。

### 扫描入库

- 会递归扫描目录下的常见视频文件扩展名。
- 扫描结果会写入 `media_items` 和 `media_files`。
- 电影当前仍是“一个视频文件对应一个 media item”。
- 剧集文件当前会聚合写入 `series`、`seasons`、`episode media_items` 和 `media_files`。
- 当前扫描会按 `(library_id, file_path)` 做增量同步。
- `mixed` 库会按单个文件判断类型：命中 `S01E02`、`1x02`、`Season 01/` 等剧集信号时按 `episode` 处理，否则默认按 `movie` 处理。
- 同路径文件会原地更新，保持原有 `media_item_id` / `media_file_id`。
- 改名或移动路径的文件会表现成旧路径删除、新路径新增。
- 已启用媒体库会启动递归 watcher，常见文件新增、删除、修改、改名和目录变化会自动触发路径级同步。
- 当本地 `.nfo`、`poster`、`fanart` 等 sidecar 文件变化时，系统会刷新同目录受影响的视频文件。
- 已启用媒体库还会定时做后台路径校准，只对齐文件路径和展示列表，避免完全依赖 watcher。
- 可以通过 `GET /api/libraries/{id}/media-items` 查询某个库下的已入库媒体条目。

### 媒体详情

- 支持查询单个媒体条目 `GET /api/media-items/{id}`。
- 支持查询某个媒体条目关联的文件列表 `GET /api/media-items/{id}/files`。
- 支持查询某个剧集条目的季列表 `GET /api/media-items/{id}/seasons`。
- 支持查询某一季下的集列表 `GET /api/seasons/{id}/episodes`。
- 支持查询某一季的海报和背景图 `GET /api/seasons/{id}/poster` / `GET /api/seasons/{id}/backdrop`。
- 支持查询剧集“全集大纲 + 本地可用集” `GET /api/media-items/{id}/episode-outline`（TMDB 可用时返回全集，缺失集可前端置灰）。
- TMDB 剧集大纲会缓存到 PostgreSQL（24h TTL），减少重复请求；缓存过期且 TMDB 失败时回退旧缓存。
- TMDB 可提供季海报和集剧照；剧照会用于集级封面/背景。TMDB 缺图时，会尝试用本地视频首帧回退（依赖 `ffmpeg`）。
- 支持管理员手动搜索单条媒体的候选元数据并应用匹配结果 `GET /api/media-items/{id}/metadata-search` / `POST /api/media-items/{id}/metadata-match`。
- 手动匹配后会持久化 TMDB 条目 ID，后续演员缓存和剧集大纲会优先走精确绑定。
- 支持手动重拉单个媒体条目元数据 `POST /api/media-items/{id}/refresh-metadata`。
- 支持查询某个媒体条目的海报和背景图 `GET /api/media-items/{id}/poster` / `GET /api/media-items/{id}/backdrop`。
- `media_files` 表里的基础字段已经可以通过 HTTP API 读出来。

### 播放链路

- 支持 `GET /api/media-files/{id}/stream` 播放媒体文件。
- 支持 `HEAD /api/media-files/{id}/stream` 读取播放相关响应头。
- 支持基础单段 `Range` 请求，可满足浏览器视频拖动和分段读取。
- 支持 `GET /api/media-files/{id}/subtitles` 查询媒体文件可切换字幕列表。
- 支持 `GET /api/subtitle-files/{id}/stream` 输出 WebVTT 字幕轨道。
- 前端播放器已支持源切换、字幕切换、缓冲状态提示、当前源重试和“从头播放”入口。
- 播放器现在会区分媒体源失败、自动播放被浏览器拦截、字幕加载失败和播放进度保存失败；字幕失败和进度保存失败都按非阻断 warning 处理，不会直接打断主视频播放。

### 播放进度

- 支持 `GET /api/media-items/{id}/playback-progress` 查询单个媒体条目的最近播放进度。
- 支持 `PUT /api/media-items/{id}/playback-progress` 更新播放位置和完成状态。
- 支持 `GET /api/playback-progress/continue-watching` 获取未看完内容的继续观看列表。
- 支持 `GET /api/watch-history` 获取当前用户自己的观看历史会话列表。
- 播放进度已经切到真实登录用户，不再挂在内部默认本地用户上。
- 播放进度、继续观看和未来用户态数据都按 `user_id` 维度隔离，不同用户互不共享。
- `playback_progress` 负责“当前最新状态”，`watch_history` 负责“一次次观看会话”的历史记录。
- 前端播放页已切到轮询上报 + 离场强制 flush，并在切源/切集前补报当前位置。

### 最小元数据解析

- 扫描时会基于文件名提取清洗后的标题。
- 常见电影文件名里的四位年份会写入 `media_items.year`。
- 扫描时会读取本地 sidecar 元数据，包括 `movie.nfo` / `<video>.nfo`、`poster.jpg`、`fanart.jpg` 等常见文件。
- 设置 `MOVA_TMDB_ACCESS_TOKEN` 后，扫描会额外查询 TMDB，补齐缺失的 `overview`、`poster_path`、`backdrop_path`、`original_title`、`year`。
- TMDB 配置当前从运行时环境变量读取，媒体库仍可单独配置元数据语言。
- 扫描命中 TMDB 图片时，会优先把海报和背景图下载到本地缓存目录，再写入媒体记录。
- `overview`、`original_title`、`sort_title`、`poster_path`、`backdrop_path` 已可通过现有详情接口返回。
- 当环境里有 `ffprobe` 时，扫描还会填充 `media_files` 的时长、编码、分辨率和码率。

### 管理端与前端测试

- 设置页已支持用户创建、编辑、删除、启停和媒体库访问授权。
- 设置页已支持媒体库扫描、删除，以及名称、描述、元数据语言和启停状态编辑，并直接展示最近一次扫描状态摘要。
- 首页库卡、首页库 shelf 和媒体库页已支持扫描中库进度条，以及条目级“发现文件 -> 补元数据 -> 获取海报简介 -> 等待写库”的连续占位卡提示；即使还没发现第一条媒体，也会先显示扫描中的 loading 状态。
- 扫描中的媒体占位卡现在会尽量保持和最终卡片接近的尺寸，减少写入媒体库完成前后的列表抖动。
- 媒体详情页现在也会感知所在媒体库的实时扫描状态：hero 区会提示当前条目或当前媒体库仍在同步；剧集详情页会把当前季相关的扫描中集卡并入 episode rail，而不是等写库后再突然出现。
- 新建启用媒体库后，同一端会先显示一个扫描中的新库占位，再由真实接口结果接管。
- 前端已接入 `Vitest + React Testing Library + jsdom` 的最小测试底座。
- 当前前端测试策略更偏向“高风险 hook + 播放器交互 + 纯函数 helper”：已覆盖实时事件 hook 的断线恢复、媒体库实时刷新、元数据实时刷新和删库跳转；扫描运行时文案与进度计算；播放器恢复进度、`from-start`、切源迁移和字幕失败降级；以及播放入口、媒体库编辑配置、设置页缓存更新这类纯函数决策。
- 后端已补上播放进度、SSE、用户 CRUD 和媒体库 CRUD 的基础测试：默认 `cargo test` 会跑纯单测；依赖 `DATABASE_URL` 的播放进度 / SSE / 用户管理 / 媒体库运行时数据库集成测试已写好，但默认标记为 `ignored`，需要在可用的 PostgreSQL 测试库上显式执行。

## 2. 当前实现边界

这些不是 bug，而是当前系统还没做完的边界：

- 剧集聚合当前依赖文件名或目录结构里出现明确的季集信号；如果用户把电影误放进 `series` 库，或者剧集文件名里没有可解析的季集号，系统可能跳过这些文件。
- 没有 `ffprobe` 或探测失败时，时长、编码、分辨率、码率等字段仍会为空。
- 当前元数据增强已支持本地 sidecar、TMDB 自动补全、TMDB 图片本地缓存、单条媒体项手动重拉，以及管理员手动重匹配；但按当前产品方向，MVP 不会把批量人工修正、字段锁定和复杂治理流程当作必需能力，缺失元数据允许保持为空。
- `watch_history` 当前只记录用户自己的观看会话；还没有单独的“已看过 / 看过几次 / 最近看完”聚合接口。
- 扫描时不会解析剧集号、季号、演员、类型等更完整的结构化元数据。
- `scan_jobs.scanned_files` 目前仍表示“已发现文件数”；后续阶段进度主要依赖 SSE 里的 `phase` 和条目级实时事件，而不是单独落库字段。
- 条目级扫描卡当前以 `item_key = file_path` 做临时身份标识，在最终写库完成前还不是数据库里的真实媒体项 ID。
- 当前只对“同路径未变化”的文件保持稳定 ID；改名或移动后的路径会被当成新文件处理。
- watcher 仍可能因为底层文件系统、Docker 挂载或网络盘行为漏掉事件，因此仍然保留手动扫描和定时路径校准作为兜底。
- 扫描任务是通过 API 进程内的 `tokio::spawn` 执行的，服务重启后任务不会恢复。当前做法是在启动时把中断任务标记为失败，允许手动重新触发。
- 当前已经有基础直链播放和字幕，也补上了错误分类、重试和非阻断降级；但还没有转码、自适应码率、音轨切换和更细的播放器兼容性治理。
- 当前用户体系已经有最小闭环，并补上了用户编辑、用户删除和密码修改；但还没有会话管理和更完整的用户管理 UI。

## 4. 距离 MVP 还差什么

如果把 MVP 定义成“管理员只负责用户和媒体库的增删改查；系统自己处理扫描和同步；所有可见用户都能感知扫描状态；普通用户可浏览、可播放、可续播；系统可在单机部署下稳定自用”，当前离 MVP 主要还差下面几块：

1. 扫描态继续前台化

- 当前已经有扫描阶段、首页库进度条和条目级占位卡，但还要继续保证“新建库后所有有权限的用户都能稳定看到扫描中”和更多页面的一致反馈。
- 重点不是任务列表，而是让系统把扫描过程自然地展示在用户已经会看的界面里。

2. 管理端进一步收口

- 设置页已经能做用户管理、媒体库扫描、媒体库删除，以及媒体库名称、描述、元数据语言和启停状态编辑。
- 按当前方向，管理员不需要关注任务内部细节，也不需要承担元数据治理；下一步更偏向把现有用户与媒体库 CRUD 继续打磨顺滑，而不是继续扩更多后台面板。

3. 播放链路最后一段稳定化

- 当前已经有字幕、切源、重试、缓冲提示、继续播放和从头播放，也补上了更清晰的错误分类与基础播放器回归测试；但还缺更多边界验证，例如不同浏览器兼容性、音轨切换和更多实际设备上的恢复路径。
- 这部分决定“能播”是否能升级成“稳定可播”。

4. 测试与文档收口

- 现在前端已有最小 Vitest，测试重点已收口到高风险 hook、播放器和纯函数 helper；后端也补上了部分数据库集成测试样例，但还缺更多扫描链路、HTTP 权限边界和设置页 CRUD 回归测试。
- MVP 形态的部署/升级说明已经单独落到 [`docs/DEPLOYMENT.md`](/Users/riches/Desktop/mova/docs/DEPLOYMENT.md)，后续主要是随着功能继续同步更新，而不是从零开始补文档。

## 5. 下一步最值得做的功能

按当前代码形态，建议优先级如下：

1. 扫描态在更多浏览入口的一致展示

- 继续把首页、媒体库页以外的浏览入口也接上更自然的扫描中提示。
- 目标是用户只看日常页面，也能感知库和条目仍在同步，不需要理解后台任务细节。

2. 剩余播放器边界与兼容性验证

- 重点放在真实浏览器里的自动播放、全屏、字幕格式兼容和异常源恢复，而不是继续堆新功能。
- 这部分更偏“把现在已经有的能力做稳”。

3. 设置页 CRUD 的前端回归测试

- 前端这边还需要把设置页里最常用的创建、编辑、删除、启停和授权路径再锁一层。
- 这样后续继续打磨管理端时，不容易把当前已经可用的最小闭环弄坏。

4. 扫描与权限边界的后端集成测试

- 继续补扫描任务 HTTP、SSE 权限过滤和更完整的数据库集成测试。
- 这样后面继续改扫描态前台化和管理端时，后端行为更不容易被静默改坏。

## 6. MVP 前最该补的测试

- 扫描任务的 HTTP 集成测试：创建、复用活跃任务、查询历史、查询单任务状态，以及阶段状态/进度字段。
- SSE 权限边界测试：确认 `scan.job.*`、`library.*`、`media_item.metadata.updated` 只推给有权限的用户。
- 条目级扫描进度测试：元数据补全和图片获取阶段的事件是否按预期推送。
- 设置页 CRUD 回归测试：创建库、编辑库、删除库、启停库，以及用户授权。
- 跨浏览器的播放器手工验证：至少覆盖自动播放限制、全屏、字幕格式和错误源重试。
